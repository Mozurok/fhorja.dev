## Adoption-signal legend
These signals are stack-agnostic. Use the package registry and compatibility axis that fit the
project's ecosystem; do not assume JavaScript.
- Registry downloads: download or install volume from the ecosystem's package registry (npm for JS/TS,
  PyPI for Python, crates.io for Rust, Go module index for Go, Maven Central for JVM, etc.). The
  primary popularity signal. State the registry and the period (for example "1.2M/week npm").
- Dependents: number of packages depending on this one in its registry (ecosystem reliance).
- Last release / release cadence: recency and rhythm of releases (maintenance liveness).
- Stars (trend): source-host stars (GitHub, GitLab) and whether the trend is rising, flat, or falling.
- Maintenance / issue health: open-issue ratio, response latency, and whether the repo is actively
  merged (active / slowing / stale).
- Framework/platform fit: compatibility on the axis that matters for THIS stack. Examples: React
  Native -> Expo support and New Architecture (Fabric / TurboModules); web -> SSR / RSC / edge runtime
  and bundle-size impact; backend -> language/runtime version range and framework integration. State
  the axis used so the column is meaningful for the stack.
- All values are a snapshot as of `Last refreshed`; treat them as directional, not durable.

# evals/

Manual eval harness for the my_work_tasks workflow. Each scenario under [`scenarios/`](./scenarios/) describes an input prompt, the expected response shape, and pass criteria you can verify by reading the model's output. Designed to be run by hand in any AI tool the workflow targets (Cursor, Claude Code, Copilot, Codex, Gemini CLI, etc.) so the same eval surface works regardless of which tool is consuming the workflow.

## Why this exists

Fhorja is a **contract**: every command's output is supposed to follow a fixed shape (frontmatter, mandatory sections, a `### Handoff` block with `Paste this next:` body, etc.). When that contract drifts (because a model regresses, a command file changes, or a shared block goes out of sync), tasks downstream break in ways that are slow to diagnose.

This eval harness catches drift before users hit it. It is not exhaustive coverage; it is a **regression net** of representative cases that exercise the load-bearing properties of the workflow.

Each scenario is self-contained: input prompt is the full text to paste into your AI tool; pass criteria are numbered and verifiable.

## Cadence

Inspired by the Anthropic guidance ("read 10-20 transcripts weekly; multi-turn evals are critical; 20-50 cases to start"):

- **Weekly (light)**: pick 1-2 scenarios, run them, read the response against the criteria. ~10 minutes. The goal is to keep contact with how the workflow performs in practice.
- **Per release (full)**: run all <!-- count:scenarios -->103<!-- /count --> scenarios before tagging a release. Failures block the release until the relevant command file is fixed (or the criteria are revised after a deliberate decision and an ADR).
- **After spec or shared-block changes**: run scenarios 01 and 03 (the most contract-sensitive ones) to confirm the change did not regress the output shape.

## How to run

Each scenario file follows the same template (see [`scenarios/template.md`](./scenarios/template.md)):

1. **Setup**: pre-existing artifacts or environment the scenario assumes (clean repo, fixture task folder, synthetic git diff, etc.). Most setups require nothing more than a fresh chat session against a checkout of this repo.
2. **Input prompt**: the exact text to paste into your AI tool. Variables like `<client>__<project>` are intentional placeholders; substitute with throwaway values.
3. **Expected response shape**: structural rules the response must satisfy (must-have sections, fence patterns, frontmatter, etc.).
4. **Pass criteria**: numbered checks you verify by reading the response. A scenario passes when every numbered check passes.
5. **Failure modes to watch**: common ways the response can pass-but-be-wrong (e.g., schema-correct but fabricated content). These are the high-value attention points.
6. **Notes**: related ADRs, related commands, known issues, history of past failures.

For a guided walkthrough that prints each scenario in turn (so you can paste from the terminal into your AI tool), run:

```bash
./evals/scripts/run-evals.sh
```

The script prints each scenario file to stdout and waits for you to press enter before continuing. It does not call any model API; the manual loop is intentional.

## What to do when a scenario fails

A failure usually means one of three things:

1. **The command file regressed**. Recent edits to `commands/<name>.md` changed the output shape. Fix: edit the command file to restore the contract, or update the scenario's pass criteria if the change was deliberate (and capture the rationale in CHANGELOG and possibly a new ADR).
2. **A shared block drifted**. The canonical block in `commands/_shared/<name>.md` changed but did not propagate to all consumers. Fix: run `./scripts/sync-shared-blocks.sh` and confirm `./scripts/lint-commands.sh` is clean.
3. **The model itself regressed**. Less common, but real (model lineup churn, sampling variance). Fix: capture the failing transcript, re-run with the most recent stable model in your tool, decide whether the regression is reproducible. If it is, file an issue.

The `### Notes` section at the bottom of each scenario file is where past failures and their resolutions are recorded. Keep adding to it; the history is a learning surface for future maintainers.

## Adding new scenarios

1. Copy [`scenarios/template.md`](./scenarios/template.md) to `scenarios/NN-short-title.md` (next free 2-digit number).
2. Fill in Goal, Setup, Input prompt, Expected response shape, Pass criteria, Failure modes, Notes.
3. Add a row to the table above.
4. Open a PR. Include rationale in the description: what gap in coverage does this fill?

Aim to grow toward Anthropic's 20-50 cases over time. Prioritize scenarios that exercise:

- **Routing edges**: cases where two commands could reasonably be the next step; the scenario validates that the right one is chosen.
- **Multi-turn flows**: the workflow's value is in sequences (`task-init` to `impact-analysis` to `decision-interview` to `implementation-plan`); single-turn scenarios miss most of the contract.
- **Adversarial inputs**: malformed, ambiguous, or context-poor prompts that the workflow should handle gracefully (often by asking a targeted question rather than guessing).

Each scenario should be **deterministic enough that two reviewers agree on whether it passed**. If the pass criterion requires judgment ("the response is clear"), refine it ("the response includes a `Reason:` line of at least 10 characters that names the specific phase").

## LLM-as-judge layer (optional second pass; ADR-0019)

Status: superseded by ADR-0033 (verify-against-rubric stateless sub-agent). The judge.py path documented below remains usable but is no longer the canonical evaluator entry point.

Slice 07 of the 2026-05-15 context-engineering uplift added `evals/scripts/judge.py`: an OPTIONAL second pass that extracts each scenario's `## Pass criteria`, formats it as a rubric, pipes the rubric to a configured AI tool, and parses per-criterion verdicts (PASS / FAIL / UNCERTAIN). The judge never replaces manual review; UNCERTAIN always defers to human; FAIL is advisory; PASS is advisory and may be spot-checked.

### Usage

```bash
# Direct invocation on a single scenario:
python3 evals/scripts/judge.py \
  --scenario evals/scenarios/01-bootstrap-and-init.md \
  --output /path/to/model-response.txt

# Stdin variant:
cat /path/to/model-response.txt | python3 evals/scripts/judge.py \
  --scenario evals/scenarios/01-bootstrap-and-init.md

# Through the harness, prompted per scenario:
./evals/scripts/run-evals.sh --judge

# Override the tool command (default: claude code --print):
./evals/scripts/run-evals.sh --judge --tool 'some-other-cli --print-only'

# JSON output (for downstream tooling):
python3 evals/scripts/judge.py --scenario ... --output ... --json
```

### Bounds and policy

- **OPTIONAL**: the judge is opt-in via `--judge`. The default loop remains fully manual.
- **Tool-agnostic**: the script uses `--tool <command>` (default `claude code --print`). Any AI tool with stdin/stdout works. No vendor API integration; no API keys handled by Fhorja.
- **Locked rubric wrapper**: the prompt the judge sends is defined as a module constant in judge.py and does not vary across runs. Changes require slice-history + ADR-0019 notes update.
- **Verdicts are advisory**: PASS/FAIL/UNCERTAIN per criterion plus an overall verdict. UNCERTAIN defers to human; FAIL is a "look closely" hint; PASS is a spot-check candidate.
- **Meta-test**: scenario 15 (`15-llm-as-judge-self-check.md`) validates the judge itself before trusting it on real outputs.

### Logging a human override of a judge verdict

ADR-0019 already says the human can override a judge verdict and should log the override in the scenario's `## History` section; this formalizes that note into one convention so a judge-vs-human agreement record accumulates over time. When a human overrides a judge verdict, add one line to the scenario's `## History` section in this exact shape:

```
YYYY-MM-DD judge-override: scenario NN, judge <PASS|FAIL|UNCERTAIN> -> human <PASS|FAIL>, reason: <one line>
```

## What this harness does **not** do

- It does not call any model API. Running scenarios is manual; you copy the input prompt into whichever AI tool you are using.
- It does not measure cost, latency, or token usage. For that, see [`scripts/measure-tokens.py`](../scripts/measure-tokens.py) and the `scripts/baseline-*.md` snapshots.
- It does not enforce that pass rates exceed a threshold; the harness is a regression net, not a benchmark suite. Failures are signals, not numbers to optimize.
- The LLM-as-judge layer is OPTIONAL and never replaces manual review (per ADR-0019; PASS/FAIL are advisory; UNCERTAIN defers to human).

If we later want a fully automated harness (paste prompts via API, capture responses, run validators programmatically), it would live alongside this manual one and would consume the same scenario files. The manual layer remains the source of truth for what each scenario means.

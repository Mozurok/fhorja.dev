# Architecture Decision Records

This directory holds the **Architecture Decision Records (ADRs)** for the Fhorja workflow operating system. ADRs document the **why** behind decisions that are otherwise only visible as code or normative content in `WORKFLOW_OPERATING_SYSTEM.md` and the command files.

Each ADR captures:

- The forces at play when the decision was made (constraints, prior incidents, conflicting goals).
- The decision itself (single, unambiguous).
- Consequences (positive, negative, neutral) accepted with the decision.
- Alternatives considered and why they were rejected.

ADRs are not a substitute for the normative spec in the WOS or the operational rules in `commands/*.md`. The spec and command files describe **what** to do; ADRs describe **why** the workflow is shaped that way. Read an ADR when you need to understand the tradeoffs behind a rule, when you are reviewing a PR that touches a load-bearing decision, or when you are considering a change that would supersede one.

## Format

Each ADR follows the [template](./template.md). New ADRs use the next available 4-digit number, in chronological order of acceptance. Once accepted, an ADR is **immutable** in spirit: superseding decisions get a new ADR (with a `Superseded by` link added to the old one), rather than rewriting history.

Status values:

- **Accepted**: the decision is in force.
- **Proposed**: the decision is under discussion; not yet enforced.
- **Deprecated**: the decision is no longer in force, but no superseding ADR exists yet.
- **Superseded by [ADR-XXXX]**: a newer ADR replaces this one; the link is the canonical successor.

## Adding a new ADR

1. Copy [`template.md`](./template.md) to `NNNN-short-title.md`, where `NNNN` is the next free 4-digit number (zero-padded).
2. Fill in Status (`Proposed` while drafting; `Accepted` only after the maintainer signs off), Date (in `YYYY-MM-DD`), and Tags.
3. Write Context, Decision, Consequences, Alternatives, References, and (optional) Notes.
4. Add a row to the Index above.
5. Open a PR. The PR description should explain why the ADR is needed now and what triggered it.

When an ADR supersedes another, add `Superseded by [ADR-XXXX](./XXXX-...)` to the old one's Status line and reference it in the new one's References section. Do not delete the old ADR; immutability is a feature, not a bug.

## When to write an ADR

Write an ADR when:

- A decision shapes the contract that consumers (commands, users, downstream tools) rely on.
- The decision has non-obvious tradeoffs that a future maintainer might not infer from the code.
- An alternative was actively considered and rejected for stated reasons.
- The decision would be expensive to undo, so the rationale should be searchable.

Do **not** write an ADR for routine changes (bug fixes, lint additions, command additions that follow existing patterns). Those belong in CHANGELOG and commit messages.
